> **Audience**: Business owners / clients testing the app manually, plus developers running
> performance, observability, and load tests.
>
> **App summary**: Reme is an AI-powered sales assistant. It remembers facts about your
> business, manages leads and appointments, runs autonomous agents (Sales, Invoice, Reminder,
> Follow-up), and surfaces insights through a memory dashboard.

## 1. Prerequisites & Environment Setup

### 1.1 Minimum Requirements

| Requirement | Version |
|---|---|
| Node.js | 20.9.0+ |
| Bun | 1.x (preferred) |
| Convex account | Free tier sufficient |
| Google AI API key | Required (default provider) |

### 1.2 Required Environment Variables

Copy `.env.example` to `.env.local` and fill in the following values before starting.

**Mandatory (app will not start without these):**

```bash
# Convex — auto-generated by `npx convex dev` on first run
NEXT_PUBLIC_CONVEX_URL=https://your-deployment.convex.cloud
CONVEX_DEPLOYMENT=your-deployment-name

# Auth
BETTER_AUTH_SECRET=your-secret-key-at-least-32-characters-long
BETTER_AUTH_URL=http://localhost:3000

# At least one AI provider key
GOOGLE_GENERATIVE_AI_API_KEY=your-google-ai-api-key

# Memory API security token (same value must also be set in Convex env)
MEMORY_API_TOKEN=your-random-secret
```

**Feature flags — all default to `false`, enable as needed:**

```bash
AI_ENABLE_MEMORY=true                # Memory retrieval in chat
AI_ENABLE_MESSAGE_PERSISTENCE=true  # Save chat history to Convex
AI_ENABLE_ANALYTICS=true            # Analytics tracking
```

**Debugging & observability (optional but recommended for testing):**

```bash
AI_DEBUG=true                              # Verbose memory/SLO logging in server console
DEBUG_MEMORY=true                          # Per-request retrieval stage latency
NEXT_PUBLIC_SHOW_CONTEXT_INSPECTOR=true    # Live memory trace panel in chat UI
```

**Observability via Langfuse (optional):**

```bash
AI_ENABLE_LANGFUSE=true
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_SECRET_KEY=sk-lf-...
LANGFUSE_HOST=https://cloud.langfuse.com
```

**SLO tuning (optional, shown with defaults):**

```bash
AI_MEMORY_RETRIEVAL_SLO_MS=1800   # Memory retrieval must complete within 1.8s
AI_CHAT_LATENCY_SLO_MS=12000      # Full chat response must complete within 12s
```

### 1.3 Set the Memory API Token in Convex

The `MEMORY_API_TOKEN` must be set in **both** `.env.local` and the Convex deployment:

```bash
npx convex env set MEMORY_API_TOKEN 'your-random-secret'
```

Generate a strong token with:

```bash
openssl rand -base64 48
```

### 1.4 Starting the App

```bash
# Install dependencies
bun install

# Start everything (Next.js + Convex dev server)
bun dev
```

The app is available at `http://localhost:3000`.

### 1.5 Developer Auth Bypass (Optional)

To skip the login screen during development:

```bash
# 1. Create a dev environment in Convex
npx convex run devSetup:createDevEnvironment '{"authUserId":"dev-user-id"}'
# Copy the returned organizationId and userId

# 2. Add to .env.local
DISABLE_AUTH_IN_DEV=true
DEV_ORGANIZATION_ID=<paste-org-id>
DEV_USER_ID=<paste-user-id>
```

Then restart the dev server. The app will skip auth entirely and use these IDs.

## 2. Authentication — Signup, Login, Logout

### 2.1 Sign Up

1. Navigate to `http://localhost:3000/register`
2. Fill in the form:
   - **Business Name** — your display name (no separate org setup step exists)
   - **Email** — use a real email format
   - **Password** — minimum 8 characters
   - **Confirm Password** — must match
3. Click **Create Account**
4. On success you are redirected to `/chat` — there is no separate onboarding wizard

**Expected result**: Chat interface loads immediately with suggestion chips visible.

**If registration fails**:
- Check that `BETTER_AUTH_SECRET` and `BETTER_AUTH_URL` are set in `.env.local`
- Check that Convex is running (`bun dev:convex`)
- Check the browser console and server logs for error details

### 2.2 Login

1. Navigate to `http://localhost:3000/login`
2. Enter your email and password
3. Click **Sign In**
4. On success you are redirected to `/chat`

### 2.3 Logout

- Click your user avatar or the logout button in the header
- You are redirected back to `/login`

### 2.4 Route Protection

Visit these routes while **logged out** — each should redirect to `/login`:

| Route | Expected |
|---|---|
| `/chat` | Redirect to `/login` |
| `/memory` | Redirect to `/login` |
| `/settings` | Redirect to `/login` |

## 5. Training the AI — Memory System

> **Prerequisite**: Enable memory in `.env.local`:
> ```bash
> AI_ENABLE_MEMORY=true
> AI_ENABLE_MESSAGE_PERSISTENCE=true
> ```
> Then restart the dev server.

### 5.1 Understanding Memory Layers

Reme has 4 memory layers, each with a different scope:

| Layer | Scope | Who Sets It |
|---|---|---|
| **Platform** | Global, all orgs | Developers (seeded data) |
| **Niche** | Per industry vertical | Developers (seeded data) |
| **Business** | Your organization | You, via chat |
| **Agent** | Per agent type, per org | Agents, via autonomous runs |

You (the user) primarily interact with the **Business** memory layer.

### 5.2 Memory Types

When storing a business memory, the AI classifies it into one of these types:

| Type | Description | Example |
|---|---|---|
| `fact` | Objective business facts | "We have 12 employees" |
| `preference` | Behavioral preferences | "Always greet leads formally" |
| `instruction` | Operational instructions | "Never discount more than 20%" |
| `context` | Background context | "We serve healthcare clients" |
| `relationship` | Information about a contact | "Sarah is the budget holder at Acme" |
| `episodic` | Past events and history | "We ran a promotion in Q1" |

### 5.3 Explicit Memory Commands

Use these phrases to explicitly teach the AI about your business:

**Store a fact:**

```
"Remember that our pricing starts at $499/month for the Starter plan."
"Remember that our business hours are Monday to Friday, 9 AM to 6 PM EST."
"Remember that our primary market is B2B SaaS companies with 50–500 employees."
```

**Store a preference:**

```
"Remember that I prefer leads to be contacted within 24 hours of inquiry."
"Remember that we never cold-call prospects — only warm outreach."
```

**Store an instruction:**

```
"Remember that invoices over $10,000 need approval from the finance team before sending."
"Remember that discounts above 15% require manager sign-off."
```

**Store relationship context:**

```
"Remember that Mark Chen at TechFlow is our champion — he's been pushing for the deal
 internally for 3 months."
```

After each command, the AI should respond confirming the memory was stored. You can verify
by visiting `/memory` and checking the Memory tab.

### 5.4 Searching Existing Memories

```
"What do you remember about our pricing?"
"Search your memory for anything about our follow-up process."
"What do you know about TechFlow?"
```

### 5.5 Updating a Memory

```
"Update: our pricing has changed. The Starter plan is now $599/month."
"Correct the business hours — we now close at 5 PM."
```

### 5.6 Deleting a Memory

```
"Forget what you know about the Q1 promotion — it's no longer relevant."
"Remove the memory about the old pricing."
```

### 5.7 Automatic Memory Extraction

In addition to explicit commands, the AI can auto-extract facts from normal conversation.
For example, if you say:

```
"We just closed a deal with BioTech Inc. for $85,000 — our biggest contract ever."
```

The AI may automatically store an episodic memory without being explicitly asked.
This depends on the context and the AI's judgment.

### 5.8 Verifying Stored Memories

After training, go to `/memory` → **Memory tab**. You should see your stored memories listed
with their type, content, and timestamp. Use the search box to filter by keyword.

## 6. Verifying Memory Retrieval

This section tests that the AI actually uses the memories you stored when answering questions.

### 6.1 Enable the Context Inspector

Add to `.env.local` and restart:

```bash
NEXT_PUBLIC_SHOW_CONTEXT_INSPECTOR=true
```

A small panel appears in the bottom-right corner of the chat. It shows:

- Which memory layers were queried
- How many memories were retrieved
- The latency of each retrieval stage
- Whether any memories exceeded the SLO threshold
- The token budget consumed per layer

### 6.2 Test Memory Recall

After storing memories (Section 5), ask questions that should trigger retrieval:

```
"What are our pricing tiers?"
"When should I follow up with a lead?"
"Tell me about our biggest client."
"How many employees do we have?"
```

**In the Context Inspector, verify:**
- Stage latency is below `AI_MEMORY_RETRIEVAL_SLO_MS` (default: 1800ms)
- Business memory layer shows retrieved memories
- The AI's answer incorporates the stored facts

### 6.3 Cross-Layer Retrieval Test

Ask a question that should pull from multiple layers:

```
"What's the best approach for following up with a lead from a healthcare company?"
```

This should retrieve:
- Platform memories (general sales best practices)
- Niche memories (healthcare-specific guidance, if seeded)
- Business memories (your follow-up preferences)

The Context Inspector should show results from multiple layers.

### 6.4 Server-Side Memory Logging

With `AI_DEBUG=true` and `DEBUG_MEMORY=true`, the server console logs detailed memory
retrieval traces for every request:

```
[Reme:Memory] Retrieval started for query: "what are our pricing tiers"
[Reme:Memory] Platform layer: 3 memories retrieved (45ms)
[Reme:Memory] Business layer: 2 memories retrieved (312ms)
[Reme:Memory] Total retrieval: 357ms (SLO: 1800ms) ✓
```

Watch the terminal running `bun dev` during chat interactions to see this output.

## 7. Autonomous Agents

Reme has 4 autonomous agents that run on Convex scheduled jobs (crons). They operate in the
background and surface their outputs via the **Agents tab** at `/memory`.

### 7.1 Agent Overview

| Agent | Trigger | Action |
|---|---|---|
| **Sales Agent** | Cron / stale leads | Monitors leads, suggests follow-ups, flags pipeline risks |
| **Invoice Agent** | Cron / overdue invoices | Tracks unpaid invoices, queues reminders |
| **Reminder Agent** | Time-based | Manages appointment follow-ups and scheduled reminders |
| **Follow-up Agent** | Post-interaction | Sequences follow-up touchpoints after meetings |

### 7.2 Observing Agent Activity

1. Navigate to `/memory` → **Agents tab**
2. The tab has two sections:
   - **Approval Queue** — pending agent actions awaiting your approval
   - **Execution Log** — completed agent runs with outcomes

### 7.3 Approval Queue Workflow

Agents with high-risk or high-impact actions require human approval before executing.

**To test the approval workflow:**

1. Create a lead that is several days old (or set a past date)
2. Wait for the Sales Agent cron to run (or trigger via Convex action in the dashboard)
3. Navigate to `/memory` → **Agents tab** → **Approval Queue**
4. You should see a pending action card with:
   - Recommended action (e.g., "Send follow-up email to John Smith")
   - Risk assessment (Low / Medium / High)
   - Countdown timer showing urgency
5. Click **Approve** to execute, or **Reject** to dismiss

**To manually trigger an agent run** (developer):

```bash
# Open the Convex dashboard
bun convex:dashboard

# Navigate to Functions → find the agent function (e.g., salesAgent:runSalesAgent)
# Run it manually from the dashboard UI
```

### 7.4 Execution Log

Each agent run is logged with:
- Timestamp
- Agent type
- Actions taken
- Outcome (success / failed / rejected)
- Memory updates made

Filter the log by agent type or date range to review historical runs.

### 7.5 Running Agent Test Scripts

## 8. Memory Dashboard Deep-Dive

Navigate to `/memory`. The dashboard has 3 tabs.

### 8.1 Memory Tab

**MemoryViewer** displays all business memories for your organization.

**Test these interactions:**

- **Search** — type a keyword in the search box; results filter in real time
- **Filter by type** — filter by `fact`, `preference`, `instruction`, `context`, `relationship`, `episodic`
- **View details** — click a memory card to expand its full content
- **Edit** — click the edit icon on a memory card to open the `MemoryEditor` modal
  - Change the content, type, or category
  - Save and verify the update appears immediately
- **Delete** — click the delete icon; confirm the deletion prompt
  - The memory should disappear from the list

**Memory health indicators to watch:**
- **Confidence score** — higher is better (0.0–1.0)
- **Access count** — how many times this memory was retrieved
- **Last accessed** — recency of use; old unused memories may decay

### 8.2 Agents Tab

See Section 7 for the Approval Queue and Execution Log walkthrough.

**Additional checks on this tab:**

- The Approval Queue badge count matches the number of pending cards
- Countdown timers on cards update in real time (no page refresh needed)
- Rejected actions disappear from the queue
- Approved actions appear in the Execution Log shortly after

### 8.3 Analytics Tab

The Analytics tab has three panels:

#### Memory Analytics

- **Total memories** — count across all types
- **Memory by type** — breakdown chart (fact / preference / instruction / etc.)
- **Retrieval rate** — how often memories are being used
- **Top retrieved memories** — which facts are most frequently surfaced

#### Agent Analytics

- **Runs per agent** — number of executions per agent type
- **Approval rate** — percentage of actions approved vs. rejected
- **Actions taken** — cumulative count of executed actions
- **Success / failure rate** — outcomes of agent runs

#### Cost Analytics

- **Token usage** — total tokens consumed (input + output)
- **Estimated cost** — based on model pricing
- **Cost by provider / tier** — breakdown if you've used multiple configurations
- **Cost trend** — daily/weekly usage graph

> **Note**: Analytics components read directly from the Convex database. The
> `AI_ENABLE_ANALYTICS` flag enables tracking of events — but the dashboard displays
> whatever data exists in Convex regardless of the flag. Enable the flag to start
> accumulating data.

## 9. Performance & Metrics

### 9.1 SLO Monitoring (Server Console)

With `AI_DEBUG=true`, every request logs SLO pass/fail to the server console:

```
[Reme:SLO] Memory retrieval: 342ms / 1800ms ✓
[Reme:SLO] Chat latency: 4.2s / 12s ✓
```

If a threshold is breached, the log line is marked with ✗ and an error level log:

```
[Reme:SLO] Memory retrieval: 2100ms / 1800ms ✗ BREACH
```

**Adjusting SLOs:**

```bash
AI_MEMORY_RETRIEVAL_SLO_MS=1200   # Tighten the memory retrieval budget
AI_CHAT_LATENCY_SLO_MS=8000       # Tighten the chat response budget
```

### 9.2 Convex Dashboard

```bash
bun convex:dashboard
```

Opens the Convex web dashboard. From here you can:

- **Monitor function calls** — see every query/mutation/action invoked with latency and errors
- **View tables** — browse `traces`, `llmUsage`, `businessMemories`, `leads`, `appointments`,
  `invoices`, `agentExecutionLogs`, and all other tables
- **Check logs** — streaming logs from all Convex functions
- **Run functions manually** — trigger any function with custom args for ad-hoc testing
- **Check scheduled jobs** — see pending cron jobs for agents

### 9.3 Convex Logs (Terminal)

```bash
bun convex:logs
```

Streams all Convex function logs to your terminal in real time. Useful for monitoring
agent runs and memory extraction without opening the web dashboard.

### 9.4 Token & Cost Tracking

Token usage is recorded in the `llmUsage` Convex table for every AI request.

**Via Convex dashboard:**
- Open the `llmUsage` table
- Each row contains: `organizationId`, `provider`, `model`, `inputTokens`, `outputTokens`,
  `estimatedCost`, `createdAt`

**Via Analytics tab:**
- `/memory` → Analytics → Cost Analytics panel shows aggregated totals and trends

### 9.5 Distributed Tracing (Convex `traces` Table)

Every AI request records spans in the `traces` table:
- `requestId` — correlates spans across a single request
- `spanName` — e.g., `memory.retrieval`, `llm.generate`, `tools.execute`
- `startTime` / `endTime` — latency per span
- `metadata` — provider, model, token counts, error details

Use the Convex dashboard to query this table and build a timeline for any request.

### 9.6 Langfuse Observability

For production-grade tracing, enable Langfuse:

```bash
AI_ENABLE_LANGFUSE=true
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_SECRET_KEY=sk-lf-...
LANGFUSE_HOST=https://cloud.langfuse.com
```

Langfuse captures:
- Full LLM call traces (prompt, response, latency, tokens)
- Tool call sequences
- Memory retrieval events
- Error traces with full context

Access your traces at `https://cloud.langfuse.com` after enabling.

### 9.7 Caching Configuration

Enable the in-process embedding cache to reduce latency on repeated queries:

```bash
AI_ENABLE_CACHING=true
AI_MEMORY_EMBEDDING_CACHE_TTL=120     # Embedding cache: 2 minutes (default)
AI_MEMORY_RETRIEVAL_CACHE_TTL=60      # Retrieval cache: 1 minute (default)
```

**Testing cache effectiveness:**
1. Send the same question twice within the cache TTL window
2. Compare retrieval latency in the Context Inspector (second request should be faster)
3. Check server logs for `[cache hit]` vs `[cache miss]` messages

### 9.8 Observability Test Script

```bash
bash scripts/test-observability.sh
```

Validates:
- Trace spans are being recorded
- Langfuse events are emitted (if configured)
- SLO thresholds are being checked
- Cost tracking entries are created

### 9.9 Memory Performance Benchmarks

| Metric | Target | SLO Threshold |
|---|---|---|
| Memory retrieval (cold) | < 800ms | 1800ms |
| Memory retrieval (cached) | < 100ms | 1800ms |
| Full chat response (simple) | < 4s | 12s |

## 11. Security Testing

### 11.1 Tenant Isolation

Each organization's data is isolated. Verify that one org cannot access another's data.

**Manual test:**

1. Register two separate accounts (different emails)
2. Store a memory in Account A: `"Remember: our secret discount is 40%"`
3. Log in as Account B
4. Ask: `"What discount do you know about?"`
5. **Expected**: Account B gets no result — it has no knowledge of Account A's memory

**Script:**

```bash
bash scripts/test-security-hardening.sh
```

Tests:
- Cross-tenant memory isolation
- API token validation on memory endpoints
- Rate limiting enforcement

### 11.2 Rate Limiting

The API has per-process rate limits (configurable via env):

```bash
AI_SECURITY_CHAT_RATE_LIMIT_PER_MINUTE=60      # Default
AI_SECURITY_APPROVAL_RATE_LIMIT_PER_MINUTE=20  # Default
```

**Manual test**: Send more than 60 chat messages in a minute.
**Expected**: The 61st request returns a `429 Too Many Requests` response.

### 11.3 Memory API Token

The Convex memory retrieval endpoints require a valid `MEMORY_API_TOKEN`.

**Test with an invalid token:**

```bash
curl -X POST http://localhost:3000/api/memory/retrieve \
  -H "Authorization: Bearer invalid-token" \
  -H "Content-Type: application/json" \
  -d '{"query": "test", "organizationId": "any-id"}'
```

**Expected**: `401 Unauthorized`

**Test with no token:**

```bash
curl -X POST http://localhost:3000/api/memory/retrieve \
  -H "Content-Type: application/json" \
  -d '{"query": "test", "organizationId": "any-id"}'
```

**Expected**: `401 Unauthorized`

### 11.4 Auth Route Protection

With no active session, all dashboard routes must redirect to `/login`:

```bash
curl -I http://localhost:3000/chat
curl -I http://localhost:3000/memory
curl -I http://localhost:3000/settings
```

**Expected**: Each returns `302` redirecting to `/login`.

### 11.5 Feedback Collection Security

```bash
bash scripts/test-feedback-collection.sh
```

Validates that feedback is properly attributed to the authenticated user's organization
and cannot be submitted by unauthenticated requests.

## 12. Known Gaps & Workarounds

| Gap | Impact | Workaround |
|---|---|---|
| **Settings not persisted to Convex** | Provider/tier selection resets on page reload | Set `AI_DEFAULT_PROVIDER` and `AI_DEFAULT_TIER` in `.env.local` |
| **No onboarding wizard** | New users land directly in chat with no guided setup | Share this guide; use suggestion chips as entry points |
| **No UI to manually trigger agents** | Cannot force an agent run from the UI | Use `bun convex:dashboard` → Functions to trigger manually |
| **`AI_ENABLE_ANALYTICS` flag** | Analytics dashboard reads Convex directly; flag only controls event tracking | Enable the flag to start accumulating data; dashboard shows existing data regardless |
| **No Playwright/Cypress** | No automated browser tests | All browser testing is manual per this guide; shell scripts cover API-level integration |
| **No dedicated onboarding flow** | No guided memory training | Use the explicit memory commands in Section 5.3 |
| **Memory decay is automatic** | Low-confidence memories are deactivated over time | Monitor via `/memory` → Memory tab; re-add important memories if they disappear |
| **Agent cron timing** | Agents run on Convex schedules — not instant | For testing, trigger manually via Convex dashboard or run test scripts |
